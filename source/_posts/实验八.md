## 1. 实验目的

1. 综合运用前七个实验所学的各类工具（虚拟机、Linux命令、SSH、SCP/rsync等），构建和模拟一个完整的网络环境；
2. 完整体验 Web 静态站点从本地开发到线上部署的全流程；
3. 掌握基于 Hexo 的静态博客系统搭建方法；
4. 理解 CI/CD（持续集成/持续部署）的核心概念，并通过 GitHub Actions 实现自动化部署；
5. 掌握在云服务器（虚拟机）上手动部署静态站点的基本操作。

## 2. 实验环境

- **本地操作系统**：Windows 11 专业工作站版 23H2
- **虚拟机环境**：Ubuntu Server LTS
- **本地开发工具**：VS Code、Git、Node.js（v18+）、Hexo CLI
- **代码托管平台**：GitHub（已注册账号）
- **Web 服务器**：Nginx（部署于 node2）
- **前置条件**：已完成实验五（虚拟机安装）、实验七（SSH远程管理），具备基本的 Linux 命令行操作能力

## 3. 实验过程

### 第一部分：基于 Hexo 的静态博客开发与 GitHub Pages 自动化部署

#### 步骤一：本地环境准备与 Hexo 初始化

由于之前实验有详细讲解，不再赘述。

#### 步骤二：创建 GitHub 仓库

登录 GitHub，创建两个仓库：

| 仓库名称 | 可见性 | 用途 |
|----------|--------|------|
| `my-blog`(自定义命名,和hexo本地项目同名) | Private（私有） | 存放 Hexo 项目源码 |
| `你的用户名.github.io` | Public（公开） | 存放最终生成的静态网页文件 |

#### 步骤三：本地项目关联到源码仓库

在 `my-blog` 目录下初始化 Git 并关联远程仓库：

```bash
git init
git remote add origin https://github.com/你的用户名/my-blog.git
```

项目里创建 `.gitignore` 文件，忽略不需要提交的目录：

```
node_modules/
.DS_Store
public/
.deploy_git/
```

提交并推送初始代码：

```bash
git add .
git commit -m "初始化 Hexo 博客源码"
git branch -M main
git push -u origin main
```

#### 步骤四：配置 GitHub Actions 自动流水线

在项目根目录下创建工作流目录和配置文件：

```bash
mkdir -p .github/workflows
nano .github/workflows/deploy.yml //默认没有nano命令，可自行去文件夹构建
```

写入以下内容（**需替换** `你的用户名` 为实际 GitHub 用户名）：

```yaml
name: Deploy to GitHub Pages

on:
  push:
    branches: [main]        # 当推送到 main 分支时触发
  workflow_dispatch:        # 支持手动触发

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: 检出源码
        uses: actions/checkout@v3

      - name: 安装 Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '20'

      - name: 缓存依赖
        uses: actions/cache@v3
        with:
          path: ~/.npm
          key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}

      - name: 安装依赖
        run: npm install

      - name: 生成静态文件
        run: npx hexo generate

      - name: 创建 .nojekyll 文件（禁用 Jekyll 处理）
        run: echo "" > public/.nojekyll

      - name: 部署到 GitHub Pages
        uses: peaceiris/actions-gh-pages@v3
        with:
          personal_token: ${{ secrets.PERSONAL_TOKEN }}
          publish_dir: ./public
          publish_branch: main
          external_repository: 你的用户名/你的用户名.github.io
```

> **![](图4：deploy.yml配置文件.png)**

#### 步骤五：配置个人访问令牌（Personal Access Token）

1. GitHub 右上角头像 → **Settings** → **Developer settings** → **Personal access tokens** → **Tokens (classic)** → **Generate new token**。
2. 勾选 `repo` 全部权限，生成并**复制**令牌值（仅显示一次）。
3. 回到源码仓库 `my-blog` → **Settings** → **Secrets and variables** → **Actions** → **New repository secret**。
   - Name：`PERSONAL_TOKEN`
   - Secret：粘贴令牌值。

进入 GitHub 仓库的 **Actions** 选项卡，查看工作流运行状态。首次运行约需 2～3 分钟，成功后访问 `https://你的用户名.github.io` 即可看到博客页面。

### 第二部分：部署到本地云服务器（虚拟机）

#### 步骤八：在服务器上安装 Nginx

选择 node2（192.168.56.103）作为 Web 服务器，通过 SSH 连接并安装 Nginx：

```bash
ssh dev@192.168.56.103
sudo apt update
sudo apt install nginx -y
sudo systemctl start nginx
sudo systemctl enable nginx
```

验证 Nginx 默认页面：sudo systemctl status nginx。或者在宿主机浏览器访问 `http://192.168.56.103`，应显示 Nginx 欢迎页。

> *（截图8：Nginx 安装完成后启用）*
> **![](图8：Nginx服务运行正常.png) ![](图8.2.png)**

#### 步骤九：创建站点目录与配置

在服务器上创建博客根目录，并调整权限：

```bash
sudo mkdir -p /var/www/blog
sudo chown -R dev:dev /var/www/blog
```

创建 Nginx 站点配置文件：

```bash
sudo nano /etc/nginx/sites-available/blog
```

写入以下内容：

```
server {
    listen 80;
    server_name _;                     # 接受所有域名/IP 访问
    root /var/www/blog;
    index index.html;
}
```

启用站点并重载 Nginx：

```bash
sudo ln -s /etc/nginx/sites-available/blog /etc/nginx/sites-enabled/
sudo nginx -t                         # 测试配置语法
sudo systemctl reload nginx
```

> *（截图9：站点配置文件及软链接）*
> **![](图9：Nginx虚拟主机配置.png)**

#### 步骤十：使用 rsync 同步静态文件

在本地（node1 或 Windows）生成 Hexo 静态文件：

```bash
cd ~/my-blog
hexo generate        # 或 hexo g，生成 public/ 目录
```

使用 rsync 将 `public/` 目录同步到服务器（node2）：

```bash
rsync -avz -e ssh --delete public/ dev@192.168.56.103:/var/www/blog/
```

参数说明：
- `-a`：归档模式
- `-v`：显示详细过程
- `-z`：传输时压缩
- `-e ssh`：使用 SSH 作为传输协议
- `--delete`：删除服务器端多余文件（保持完全一致）

> *（截图10：rsync 同步过程，过长只截取部分）*
> **![](图10：rsync增量同步部署.png)**

在宿主机浏览器访问 `http://192.168.56.103`，应显示与 GitHub Pages 一致的博客页面。至此，完成了从源码到两种部署环境（云端 Pages 和自建服务器）的完整交付。

## 5. 知识总结

### 5.1 CI/CD 核心概念

| 概念 | 含义 | 本实验体现 |
|------|------|------------|
| **持续集成（CI）** | 代码变更后自动执行构建、测试 | `git push` 触发 Actions 自动运行 `hexo generate` |
| **持续部署（CD）** | 构建通过后自动将制品部署到生产环境 | Actions 将 `public/` 推送到 GitHub Pages |
| **持续交付** | 确保代码随时可部署（手动或自动） | 通过工作流保持站点与源码同步 |

### 5.2 整体部署架构（三层链路）

| 层级 | 组件 | 作用 |
|------|------|------|
| **开发环境** | VS Code + Hexo + Git | 编写 Markdown 文章，生成静态站点 |
| **CI/CD 流水线** | GitHub Actions | 自动安装依赖、构建、上传部署 |
| **生产环境** | GitHub Pages / Nginx | 托管静态文件，对外提供服务 |

### 5.3 Hexo 常用命令速查

| 命令 | 说明 |
|------|------|
| `hexo init [folder]` | 初始化博客项目 |
| `hexo new [title]` | 创建新文章 |
| `hexo generate` / `hexo g` | 生成静态文件到 `public/` |
| `hexo server` / `hexo s` | 本地预览（http://localhost:4000） |
| `hexo clean` | 清理缓存和 `public/` 目录 |
| `hexo deploy` | 部署到配置的远程仓库（本实验未使用） |

### 5.4 GitHub Actions 工作流核心要素

| 要素 | 说明 |
|------|------|
| `on.push.branches` | 触发条件：指定分支的 push 事件 |
| `jobs` | 定义作业（一个或多个） |
| `runs-on` | 运行环境（如 `ubuntu-latest`） |
| `steps` | 具体步骤，可调用官方或社区 Action |
| `secrets` | 存储敏感信息（如 Personal Token） |
| `actions/cache` | 缓存依赖，加速后续构建 |

### 5.5 部署方式对比

| 部署方式 | 优点 | 缺点 | 适用场景 |
|----------|------|------|----------|
| **GitHub Pages + Actions** | 全自动、免费、支持 CDN | 仅支持静态站点 | 个人博客、开源文档 |
| **Nginx + rsync** | 服务器可控、支持动态扩展 | 手动同步、需维护服务器 | 企业内网、私有部署 |
| **传统 FTP/SCP** | 简单 | 重复劳动、易出错 | 已不推荐 |


## 6. 出现问题及解决方案

| 问题现象 | 可能原因 | 解决方法 | 是否解决 |
|----------|----------|----------|----------|
| `hexo: command not found` | Node.js 未安装或 Hexo 未全局安装 | 安装 Node.js 并执行 `npm install -g hexo-cli`或者每个命令前加npx | ✅ |
| GitHub Actions 报错 `Permission denied to username.github.io` | 未配置 Personal Token 或 Token 权限不足 | 重新生成 Token 并勾选 `repo` 权限，添加至仓库 Secrets | ✅ |
| 部署后页面样式丢失 | Hexo 配置中 `url` 未正确设置 | 修改 `_config.yml` 中 `url: https://用户名.github.io` | ✅ |
| 服务器端访问显示 Nginx 默认页，而非博客 | 默认站点优先级高于新建站点 | 删除 `/etc/nginx/sites-enabled/default` 或修改默认 server 的 root | ✅ |
| rsync 同步时提示 `Permission denied (publickey)` | 本地未配置 SSH 免密登录到服务器 | 执行 `ssh-copy-id dev@192.168.56.103` 配置密钥 | ✅ |
| 修改文章后 GitHub Pages 未更新 | Actions 工作流未触发（如 push 到非 main 分支） | 检查分支名称；或手动触发 `workflow_dispatch` | ✅ |


## 7. 心得体会

### 技术层面

通过本次实验，我完整经历了从静态博客开发、版本管理、自动化部署到手动运维的全流程。**Hexo** 作为一个高效的静态站点生成器，将 Markdown 写作与 HTML 生成无缝结合，让我体会到了“内容与样式分离”的便捷。**GitHub Actions** 的自动化流水线展现了 CI/CD 的强大魅力——一次配置，后续每次提交都能自动完成构建与部署，极大地减少了重复劳动。而手动在服务器上安装 Nginx 并使用 rsync 同步，则让我更深入地理解了传统运维的基本操作，两种方式互为补充，使我对 Web 部署有了立体的认知。

 **缓存机制** 对构建速度的提升——`actions/cache` 缓存了 `node_modules`，使第二次构建仅需几十秒，这在真实生产环境中意义重大。同时，`rsync --delete` 保证了服务器端与本地严格一致，避免了“垃圾文件”堆积问题。

### 思维层面

CI/CD 理念体现了 **自动化优先** 和 **流水线化** 的工程思想，它将人为规则转化为标准化流程，降低了交付风险，使开发者能专注于业务逻辑。GitHub Actions 作为开源社区的基础设施，让个人开发者也能享受企业级的 DevOps 工具链，这正是开源精神“普惠”的体现。

在对比 GitHub Pages 与自建服务器时，我认识到：没有最好的方案，只有最合适的场景。对个人博客，云托管（Pages）是成本最低、性能最优的选择；对企业内网或需后端接口的项目，则自建服务器更灵活。这种辩证思考也启发我，在实际工作中要综合评估需求、成本、维护成本等因素，做出合理的架构决策。

此外，在配置 Token、管理 Secrets 的过程中，我进一步体会到了 **安全底线** 的重要性——Token 一旦泄露，攻击者可完全控制仓库，因此必须妥善保管、定期更换。这不仅是技术操作，更是信息时代每位开发者应具备的基本素养。
