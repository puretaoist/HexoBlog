## 1. 实验目的

本次实验旨在安装和配置 OpenSSH 服务，实现多台虚拟机之间的远程登录管理；配置 SSH 密钥认证，实现免密登录，理解公钥/私钥对的身份验证原理；掌握使用 SCP 和 rsync 进行文件传输与目录同步的方法；同时了解多机并发环境下的账户管理策略，为后续集群管理与自动化运维奠定基础。

## 2. 实验环境

- **操作系统**：Ubuntu Server LTS（node1、node2）
- **运行载体**：VirtualBox 7.0.12 虚拟机（继承实验五的集群环境）
- **网络**：Host-Only 网络，IP 段 192.168.56.1/24
- **节点信息**：
  - node1：192.168.56.102（主操作机）
  - node2：192.168.56.103（远程目标机）
  - 截图为个人实验，可能会与教程有命名出入
- **用户**：所有节点均存在 `dev` 用户（需提前创建，或使用自定义用户）
- **前置条件**：所有节点网络互通，已安装 OpenSSH 客户端（默认已安装）

## 3. 实验过程

### 第一部分：SSH 服务安装与基础配置

#### 步骤一：检查并安装 SSH 服务

在 node1 和 node2 上分别检查 SSH 服务状态，若未安装则进行安装：

```bash
# 检查 SSH 服务状态
sudo systemctl status ssh

# 若未安装，执行安装
sudo apt update
sudo apt install openssh-server -y

# 设置开机自启
sudo systemctl enable ssh
sudo systemctl start ssh

# 确认监听端口 22
sudo ss -tlnp | grep ssh
```

> *（截图1：安装 openssh-server 及服务启动状态）*
> **![](图1.png)**


#### 步骤二：配置 SSH 服务器（安全加固）

编辑 `/etc/ssh/sshd_config` 配置文件，进行安全基础设置：

```bash
sudo nano /etc/ssh/sshd_config
```

修改或确认以下关键配置项：

```bash
Port 22                     # 默认端口（可改为其他端口增强安全，本次保持默认）
PermitRootLogin no          # 禁止 root 直接登录（重要安全措施）
PubkeyAuthentication yes    # 允许公钥认证
PasswordAuthentication yes  # 允许密码认证（保留，便于初次连接）
ChallengeResponseAuthentication no   # 禁用挑战响应认证
UsePAM yes                  # 启用 PAM 模块
```

保存退出后重启服务：

```bash
sudo systemctl restart ssh
```

### 第二部分：SSH 密钥认证配置

#### 步骤三：生成 SSH 密钥对（在 node1 上）

在 node1 上，以 `dev` 用户身份（或 自定义）生成 ED25519 密钥对（更安全、速度更快）：

```bash
# 切换到你的用户（若不存在则先创建）
sudo adduser dev
sudo usermod -aG sudo dev
su - dev

# 生成密钥对，-t 指定算法，-C 添加注释（邮箱）
ssh-keygen -t ed25519 -C "你的邮箱@example.com"

# 一路回车，使用默认路径（~/.ssh/id_ed25519）且不设置密码短语（免密登录）
```

生成后查看文件：

```bash
ls -l ~/.ssh/
# 应看到 id_ed25519（私钥）和 id_ed25519.pub（公钥）
cat ~/.ssh/id_ed25519.pub   # 查看公钥内容
```

> *（截图3：ssh-keygen 生成密钥对过程）*
> **![](图3.png)**


#### 步骤四：将公钥复制到远程服务器（node2）

**方法一（推荐）：使用 ssh-copy-id**

在 node1 上执行：

```bash
# 将公钥复制到 node2 的 dev 用户下
ssh-copy-id dev@192.168.56.103
```

根据提示输入 node2 上 `dev` 用户的密码，完成后公钥将自动追加到 node2 的 `~/.ssh/authorized_keys` 文件中。(注意：可能需要node2虚拟机启动才能正常复制)

**方法二（手动，备用）**：

```bash
# 在 node1 上显示公钥内容
cat ~/.ssh/id_ed25519.pub

# 在 node2 上，以 dev 用户执行：
mkdir -p ~/.ssh
echo "公钥内容" >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys
chmod 700 ~/.ssh
```

#### 步骤五：测试免密登录

在 node1 上，使用 `ssh` 命令连接 node2，验证无需密码：

# 执行远程命令测试

### 第三部分：文件传输与同步

#### 步骤六：使用 SCP 传输文件

SCP（Secure Copy）基于 SSH，用于加密传输文件或目录。

**准备工作**：在 node1 上创建测试文件：

```bash
echo "Hello from node1" > test.txt
mkdir project
echo "file1" > project/file1.txt
echo "file2" > project/file2.txt
```

**SCP 操作示例**：

```bash
# 1. 上传单个文件到 node2 的家目录
scp test.txt dev@192.168.56.103:/home/dev/
(dev为用户名)

# 2. 上传整个目录（-r 递归）
scp -r project/ dev@192.168.56.103:/home/dev/

# 3. 从 node2 下载文件到当前目录
scp dev@192.168.56.103:/home/dev/test.txt ./downloaded_test.txt

# 4. 从 node2 下载整个目录
scp -r dev@192.168.56.103:/home/dev/project/ ./downloaded_project/
```

验证文件是否传输成功：

```bash
# 在 node2 上查看
ssh dev@192.168.56.103 "ls -l /home/dev/"
```

#### 步骤七：使用 rsync 增量同步

rsync 支持增量同步，只传输变化的部分，适合大规模目录同步和备份。

**安装 rsync**（所有节点）：

```bash
sudo apt install rsync -y
```

**本地同步示例**：

```bash
# 在 node1 上创建源目录和目标目录
mkdir -p source_data backup
echo "data1" > source_data/data1.txt
echo "data2" > source_data/data2.txt

# 同步源目录到备份目录（-a 归档，-v 详细，-z 压缩传输）
rsync -avz source_data/ backup/

# 修改源文件后再次同步（只传输变更部分）
echo "append" >> source_data/data1.txt
rsync -avz source_data/ backup/
```

**远程同步示例**：

```bash
# 1. 推送本地 project 目录到 node2（使用 -e ssh 指定传输协议）
rsync -avz -e ssh project/ dev@192.168.56.103:/home/dev/project/

# 2. 从 node2 拉取 logs 目录到本地
mkdir logs
rsync -avz -e ssh dev@192.168.56.103:/home/dev/logs/ ./logs/

# 3. 保持两端完全一致（--delete 删除目标端多余文件）
# 先在 node1 的 source_data 中删除 data2.txt，再同步
rm source_data/data2.txt
rsync -avz --delete -e ssh source_data/ dev@192.168.56.103:/home/dev/source_data/
# 此时 node2 上的 source_data 也会删除 data2.txt
```

**高级过滤同步**：

```bash
# 排除特定目录或文件
rsync -avz --exclude='*.log' --exclude='tmp/' -e ssh source_data/ dev@192.168.56.103:/home/dev/source_data/
```

### 第四部分（扩展）：SSH 配置文件简化连接

为简化重复输入用户和 IP，可在 `~/.ssh/config` 中配置别名。

```bash
nano ~/.ssh/config
```

添加内容：

```
Host node2
    HostName 192.168.56.103
    User dev
    Port 22
    IdentityFile ~/.ssh/id_ed25519

Host node3
    HostName 192.168.56.104
    User dev
    Port 22
    IdentityFile ~/.ssh/id_ed25519
```

之后即可使用 `ssh node2` 快捷登录。

## 5. 知识总结

### 5.1 SSH 认证方式对比

| 认证方式 | 原理 | 安全性 | 适用场景 | 配置要点 |
|----------|------|--------|----------|----------|
| **密码认证** | 用户输入密码，服务器验证 `/etc/shadow` | 较低（易被暴力破解） | 临时访问、首次连接 | `PasswordAuthentication yes` |
| **密钥认证** | 客户端用私钥签名，服务器用公钥验证 | 高（无需传输密码） | 日常管理、自动化脚本 | 生成密钥对，公钥放入 `~/.ssh/authorized_keys`，`PubkeyAuthentication yes` |

### 5.2 SSH 密钥对管理要点

- **私钥**（`id_ed25519`）严格保密，权限应为 `600`，不可泄露。
- **公钥**（`id_ed25519.pub`）可公开，添加到远程主机的 `authorized_keys` 文件中。
- **`authorized_keys` 文件权限**必须为 `600`，`.ssh` 目录权限为 `700`，否则 SSH 会拒绝使用。
- 可使用 `ssh-copy-id` 自动复制公钥，避免手动操作错误。
- 推荐使用 ED25519 算法（`-t ed25519`），比 RSA 更短更安全。

### 5.3 SCP vs rsync 对比

| 工具 | 特点 | 优缺点 | 适用场景 |
|------|------|--------|----------|
| **SCP** | 基于 SSH，加密传输；每次全量复制 | 简单易用，但不支持增量；传输大目录重复数据多 | 少量文件一次性传输 |
| **rsync** | 增量传输（只传差异）；支持压缩、删除、过滤 | 效率高，支持断点续传（需结合--partial）；命令复杂 | 目录同步、定期备份、持续部署 |

**rsync 常用参数**：

| 参数 | 作用 |
|------|------|
| `-a` | 归档模式（保留权限、时间戳、递归等） |
| `-v` | 详细输出 |
| `-z` | 传输时压缩 |
| `-e ssh` | 指定使用 SSH 作为传输通道 |
| `--delete` | 删除目标端多余文件（保持完全一致） |
| `--exclude` | 排除指定文件/目录 |
| `--partial` | 保留部分传输的文件，便于断点续传 |

### 5.4 SSH 安全加固建议

- 禁用 root 登录：`PermitRootLogin no`
- 更改默认端口（如 2222），降低被扫描风险
- 仅允许密钥登录（`PasswordAuthentication no`），但本次实验保留密码以便演示
- 使用 `fail2ban` 防止暴力破解
- 定期更新 OpenSSH 版本

### 5.5 常用 SSH 命令速查

| 命令 | 说明 |
|------|------|
| `ssh user@host` | 远程登录 |
| `ssh user@host "command"` | 远程执行单条命令 |
| `ssh-copy-id user@host` | 复制公钥到远程 |
| `scp file user@host:/path/` | 上传文件 |
| `scp -r dir user@host:/path/` | 上传目录 |
| `scp user@host:/path/file ./` | 下载文件 |
| `rsync -avz -e ssh src/ user@host:/dst/` | 远程增量同步推送 |
| `rsync -avz -e ssh user@host:/src/ ./dst/` | 远程增量同步拉取 |


## 6. 出现问题及解决方案

| 问题现象 | 可能原因 | 解决方法 | 是否解决 |
|----------|----------|----------|----------|
| 执行 `ssh-copy-id` 提示 `Permission denied` | 远程用户密码错误或 SSH 服务未允许密码认证 | 确认密码正确；检查远程 `/etc/ssh/sshd_config` 中 `PasswordAuthentication yes` 并重启服务 | ✅ |
| 免密登录仍提示输入密码 | 公钥未正确添加，或 `authorized_keys` 权限不对 | 检查 `~/.ssh/authorized_keys` 内容是否正确；执行 `chmod 600 ~/.ssh/authorized_keys`；检查 `.ssh` 目录权限为 700 | ✅ |
| SCP 传输目录时提示 `not a regular file` | 未使用 `-r` 参数 | 使用 `scp -r` 递归传输目录 | ✅ |
| rsync 远程同步失败，提示 `command not found` | 远程主机未安装 rsync | 在远程主机执行 `sudo apt install rsync -y` | ✅ |
| rsync 同步后文件权限/时间戳变化 | 未使用 `-a` 归档选项 | 使用 `rsync -avz` 保留属性和时间 | ✅ |
| 使用 `ssh node2` 提示 `Could not resolve hostname` | 未配置 `/etc/hosts` 或 `~/.ssh/config` | 在 hosts 中添加解析（如 `192.168.56.103 node2`），或正确配置 config 文件 | ✅ |
| 生成密钥时设置密码短语，导致免密失效 | 密钥加密需要输入密码 | 重新生成密钥，不设置密码短语（直接回车）；若已生成可执行 `ssh-add` 添加私钥到代理 | ✅ |
| 公钥复制到多台机器后，某台仍要求密码 | 目标机器的 `~/.ssh/authorized_keys` 末尾缺少换行 | 确保公钥内容单独一行，且末尾有换行符 | ✅ |


## 7. 心得体会

### 技术层面

本次实验让我深入理解了 SSH 协议在远程管理中的核心地位。通过配置密钥认证，我认识到了公钥/私钥体制的便利性与安全性——一次配置，后续所有操作（登录、SCP、rsync）均可免密进行，极大提升了工作效率。实验过程中，我掌握了 `ssh-copy-id` 的便捷用法，理解了 `authorized_keys` 文件的权限要求，这些细节在实际生产环境中至关重要。

SCP 和 rsync 的对比让我认识到，**增量同步**在数据量较大时具有明显的性能优势。rsync 的 `--delete` 参数能够保持两端完全一致，非常适合用于备份和部署场景。同时，我也体会到命令行工具的灵活性——通过参数组合可以满足各种复杂需求（如排除特定文件、压缩传输等）。

此外，SSH 配置文件 `~/.ssh/config` 的别名功能极大简化了日常操作，将冗长的 `ssh dev@192.168.56.103` 简化为 `ssh node2`，体现了 Linux 工具追求高效的设计理念。

### 思维层面

SSH 作为开源网络协议，其广泛应用体现了互联网协作的安全基石。在当今网络安全形势日益严峻的背景下，掌握加密通信和身份认证技术是每位开发运维人员的必备素养。实验中我们禁止 root 登录、使用密钥认证，这些小细节体现了 **最小权限原则** 和 **纵深防御** 的安全思想。

同时，rsync 和 SCP 等工具的开源属性，使得开发者能够自由使用并贡献改进，这种开放共享的精神推动了云计算和自动化运维的高速发展。应理解其背后的安全模型和设计哲学，在实际工作中做到 **便捷与安全并重**，为构建稳定可靠的信息系统贡献力量。
